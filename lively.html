Lively

A statically typed procedural language built from scratch — lexer, parser, three-address code, bytecode compiler, and a stack-based virtual machine.

Role Solo · Designer & Implementer
Language C++17
Status In active development
Repo github.com/KanishkSigar/Lively ↗
01 — Motivation

Why build a language?

Most developers use programming languages every day without ever seeing what happens between hitting run and getting an answer. Lively started as a way to close that gap — to build, end-to-end, every stage of a working language and understand each layer well enough to defend my choices.

The goal wasn't a production language. It was to ship a small, statically typed procedural language with the full pipeline: source → tokens → AST → typed IR → three-address code → bytecode → running program. If a textbook talks about it, Lively has a version of it.

Each stage is small, isolated, and replaceable. The boundaries between stages are the project's most important interfaces — they're what made it possible to ship without rewriting earlier layers every time I learned something new.

03 — The language

What Lively looks like

Syntax is procedural and familiar — closer to a C-family language than to anything functional. Static types, explicit returns, no hidden coercions. If it compiles, the type checker has already ruled out an entire class of runtime failure.

examples/fib.lv 
fn fib(n: int) -> int {
  if n < 2 { return n; }
  return fib(n - 1) + fib(n - 2);
}

fn main() -> int {
  let i: int = 0;
  while i < 10 {
    print(fib(i));
    i = i + 1;
  }
  return 0;
}
04 — Lowering

From AST to instructions

Direct AST-walking interpreters are easy to write but slow and hard to extend. Lively lowers the AST into a three-address code (TAC) form first — explicit temporaries, one operation per statement — and then compiles that into a flat bytecode the VM can run.

TAC
// fib(n) lowered
t0 = n < 2
if t0 goto L_ret
t1 = n - 1
t2 = call fib, t1
t3 = n - 2
t4 = call fib, t3
t5 = t2 + t4
return t5
L_ret:
return n
Bytecode
// stack-based ops
LOAD_LOCAL  n
PUSH_CONST  2
LT
JMP_IF_TRUE L_ret
LOAD_LOCAL  n
PUSH_CONST  1
SUB
CALL        fib, 1
LOAD_LOCAL  n
PUSH_CONST  2
SUB
CALL        fib, 1
ADD
RET
05 — Virtual machine

A small stack machine

The runtime is a stack-based VM with a value stack, a call stack, and a tiny opcode table. Operands push onto the stack, operators pop and push, control flow is a few jump variants. Function calls allocate a frame, locals live on the value stack, the frame pointer walks up and down — the same playbook every textbook VM uses, written from scratch in C++.

  • Compact opcodes. Fixed-size instruction stream, no per-op heap allocations on the hot path.
  • Typed values. Tagged union for the value type — clean dispatch, no virtuals on the inner loop.
  • Function frames. Locals indexed off the frame pointer, return addresses tracked in a separate call stack.
  • Debug hooks. Stack trace, op disassembly, and a step mode for tracing execution one instruction at a time.
06 — Lessons

What I learned the hard way

01

Boundaries between stages are everything.

The first version mashed parsing and type-checking together. Untangling them was painful — and the moment I did, every later change got easier. The TAC layer exists for the same reason: it gives the compiler a clean place to optimise without touching the parser.

02

Error messages are part of the language.

A language with bad errors feels hostile. I spent more time on parser error recovery and type-error formatting than on the VM. It's the difference between a toy and something I'd actually want to write code in.

03

Premature optimisation kills momentum.

I almost went down the rabbit hole of register allocation before the language could even call functions. Shipping the slow-but-correct stack VM first turned out to be the right call — a working baseline beats half a fast thing.

07 — What's next

Where Lively goes from here

  • A register-based VM as an alternative backend, to compare performance head-to-head with the stack VM.
  • First-class functions and a small closure model.
  • A standard library worth using — strings, collections, basic I/O.
  • A REPL with incremental compilation for faster iteration.
